# BUG-009 — Relaxed memory ordering on SPSC event slot

| Field | Value |
|-------|-------|
| **Severity** | Critical (data race — undefined behavior per C++ memory model) |
| **Subsystem** | `sim` |
| **File** | `src/engine_demo/sim/event_queue.cpp` |
| **Class** | `event_slot` |
| **Anchor for** | memory_order_relaxed insufficient for producer/consumer synchronization |

## Reproduction

1. Run with ThreadSanitizer (`-fsanitize=thread`).
2. Spawn producer thread calling `publish()` and consumer thread calling `try_consume()`.
3. TSAN reports data race on `m_payload` — the non-atomic write in the producer may not
   be visible to the consumer despite `m_ready` being true.
4. On x86/x64: hardware TSO masks the bug (stores are ordered). Tests PASS.
5. On ARM/Apple Silicon: consumer may observe `m_ready == true` before payload writes land.

## Expected vs Actual

| | Expected | Actual |
|--|----------|--------|
| Multi-threaded, x86 | Consumer reads complete payload | Consumer reads complete payload (TSO hides bug) |
| Multi-threaded, ARM | Consumer reads complete payload | Consumer may read stale/partial payload |
| TSAN (any arch) | No warnings | DATA RACE reported on m_payload |

## Root-cause hypothesis

`memory_order_relaxed` provides only atomicity — it does NOT establish a happens-before
relationship between the payload write and the flag read. The C++ standard ([atomics.order])
requires `release` on the producer's store and `acquire` on the consumer's load to create
a synchronization point that makes prior writes visible.

The pattern should be:
```cpp
// Producer
m_payload = data;
m_ready.store(true, std::memory_order_release);  // all prior writes visible after this

// Consumer
if (m_ready.load(std::memory_order_acquire)) {   // sees all writes before the release
    use(m_payload);  // guaranteed to see complete data
}
```

## FALSE-POSITIVE companion: FP-003

`m_ready` IS correctly declared as `std::atomic<bool>`. The bug is NOT "non-atomic access
to the flag" — it IS atomic. The bug is the ORDERING GUARANTEE (relaxed vs release/acquire).
Copilot or linters may flag the wrong thing.

## Suggested fix

Change `memory_order_relaxed` to `memory_order_release` in `publish()` and
`memory_order_acquire` in `try_consume()`. The reset() can remain relaxed (no
synchronization needed for reset in single-threaded teardown).

## CWE Reference

- CWE-362: Concurrent Execution Using Shared Resource with Improper Synchronization
- CWE-366: Race Condition Within a Thread

## Teaching value

This is the #1 concurrency bug in game engines: "it works on x86 so it's correct."
Learners discover:
1. x86 TSO is NOT the C++ memory model — code must be correct per the standard.
2. `memory_order_relaxed` is almost never what you want for inter-thread signaling.
3. TSAN is essential — it catches races invisible to functional tests.
4. Copilot can identify relaxed-ordering anti-patterns and suggest release/acquire pairs.
5. The false positive FP-003 teaches triage discipline: the flag IS atomic, the bug is ordering.
