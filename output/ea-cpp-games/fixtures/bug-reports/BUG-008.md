# BUG-008 — memcmp on struct with padding bytes

| Field | Value |
|-------|-------|
| **Severity** | Medium (intermittent false negatives in replay validation) |
| **Subsystem** | `sim` |
| **File** | `src/engine_demo/sim/replay_state.cpp` |
| **Class** | N/A (free function `snapshot_equals`) |
| **Anchor for** | memcmp compares uninitialized padding → non-deterministic comparison |

## Reproduction

1. Construct two `replay_snapshot` instances independently (not via copy/assignment).
2. Set all member fields to identical values in both.
3. Call `snapshot_equals(a, b)`.
4. Observe: sometimes returns `false` even though all logical fields are equal.
5. The result depends on prior stack contents (what garbage was in the padding bytes).

## Expected vs Actual

| | Expected | Actual |
|--|----------|--------|
| Independently-constructed, same values | `snapshot_equals` returns true | May return false (padding differs) |
| Copied via `=` | `snapshot_equals` returns true | Returns true (copy includes padding) |

## Root-cause hypothesis

`replay_snapshot` has 11 bytes of padding (due to alignment of `double` members):
- 7 bytes after `flags` (uint8_t at offset 0, next double at offset 8)
- 4 bytes after `frame_number` (uint32_t at offset 24, next double at offset 32)

`make_snapshot()` assigns members individually but does NOT zero-initialize the struct
first. The padding bytes retain whatever was on the stack. Two snapshots constructed in
different stack frames will have different padding → `memcmp` reports mismatch.

## Suggested fix

Option A (recommended): Compare members explicitly:
```cpp
return a.flags == b.flags
    && a.position_x == b.position_x
    && a.position_y == b.position_y
    && a.frame_number == b.frame_number
    && a.velocity_x == b.velocity_x
    && a.velocity_y == b.velocity_y;
```

Option B: Zero-initialize with `= {}` or `std::memset(&snap, 0, sizeof(snap))` before
assigning members.

Option C: Reorder members to eliminate padding (doubles first, then uint32_t, then uint8_t).

## CWE Reference

- CWE-188: Reliance on Data/Memory Layout

## Teaching value

This is a classic "works on my machine" bug — it depends on stack state. Learners discover:
1. `std::has_unique_object_representations_v<T>` tells you if memcmp is safe.
2. struct layout rules and `sizeof()` vs sum-of-members.
3. Copilot can identify padding issues and suggest member-wise comparison.
4. `-Wpadded` compiler flag can alert on structs with padding.
